Git in Your Workflow

Keep that command line open

One of the hardest things for Git newcomers is figuring out how Git fits with their workflow.

Everyone knows that you have to save files on a computer. Whether its an image or a document, File > Save is a routine thing we do to save our work.

Git adds a couple of extra steps to the process of saving work.

The basic Git-powered workflow looks something like this:

  1. Make some changes to your file
  2. Save your file
  3. Tell Git about the file or files you want to commit with git add
  4. Tell Git to preserve a snapshot of those files with git commit
  5. Make some more changes to your file...then add, the commit

In short, you change your file, you save the file, you add it to a list of files ready for preservation, and then you commit it to your repository. Change. Save. Add. Commit.

Changing & Saving Files

Git works best with plain text formats (such as HTML or CSS files), but it can track changes to anything, including image files.

Just as you normally would, you make changes to your files, and you save them when the work is good. But if you get to a point where you have another idea or an alternative approach you want to try out, it's time to tell Git about your changes.

Tell Git What to Commit

Sometimes a meaningful change that you want to commit is contained in a single file. If you're working on a Web page called about.html, and you revise a paragraph of copy, you just need to commit that one file: 

$ git add about.html

Once you add the file, git status will tell you what files the commit will include: 

$ git status
# On branch master
# Changes to be committed:
#   (use "git reset HEAD <file>..." to unstage)
#
#	modified:   index.html
#

Sometimes, though, one change is spread over two different files. Imagine that you revised a class in HTML, like <h1 class="page_title">, to a simpler <h1 class="title">. You would also need to adjust your CSS selector from h1.page_title to h1.title.

In that particular case, you edited two different files, but for a single change. Both files should be staged for a commit: git add index.html css/styles.css

Tell Git to Commit

Once you have checked via $ git status that the correct file(s) are staged for commit, it's time to make the commit. The shorthand version for this is $ git commit -m "Commit message goes here": 

$ git status
# On branch master
# Changes to be committed:
#   (use "git reset HEAD <file>..." to unstage)
#
#	modified:   index.html
# modified:   css/styles.css
#

$ git commit -m "Changed .page_title to simply .title"
[master 3df6a79] Changed .page_title to simply .title
 2 files changed, 2 insertions(+), 2 deletions(-)

Note that in Git, a changed line will be reported as both an insertion, and a deletion. From a certain perspective, this is true: you "deleted" the old line, and "inserted" a new one (even if it was only part of the line that changed.
